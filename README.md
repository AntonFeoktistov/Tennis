# 🎾 Tennis 

Проект предоставляет функционал по регистрации нового теннисного матча, обновлению таблицы матча, выявлению победителя, просмотру списка текущих и завершенных матчей, поиску матчей по имени участников.

Обновление счета матча осуществляется по нажатию кнопки, соответствующей игроку. Добавление очков происходит по стандартным теннисным правилам. Побеждает тот, кто первым набрал 2 сета.

---

## 🏗️ Архитектура проекта

### **Main, Router, Container**

Запуск сервера осуществляется из файла `main.py`. При этом незавершенные матчи автоматически загружаются в атрибут cache сервиса. Инициализируются container и router.

Класс `Container` представляет собой фабрику по производству контроллеров: отдельного на каждую из 4 страниц проекта. `Router` в зависимости от пути, указанного в запросе, перенаправляет запрос из базового хендлера в хендлер страницы, вызывая у того соответствующий метод (GET/POST).

### **Controller**

Первичная обработка запросов осуществляется в классе `BaseHandler`, который перенаправляет запрос через роутер в постраничные контроллеры, которые наследуются от абстрактного `BaseController`.

- **IndexController** — отвечает за выдачу главной страницы
- **NewMatchController** — выдает страницу регистрации нового матча, перенаправляет на страницу с таблицей нового матча
- **MatchScoreController** — отвечает за страницу с таблицей счета, обрабатывает запросы на добавление очков
- **MatchesController** — обрабатывает запросы на показ списка всех матчей

### **View**

В модуле осуществляется взаимодействие с шаблонизатором — рендеринг страниц согласно данному контексту. Шаблоны страниц помещаются в каталоге `templates` базового каталога, css — в каталоге `static` базового каталога.

### **Dto**

Модуль содержит описание dataclass-ов для объектов response, request, а также для объектов, описывающих игроков, матчи, состояние счета.

### **Service**

Модуль осуществляет обработку запроса из контроллера, создание новых матчей, обновление таблиц счета текущих матчей, добавление завершенных матчей в бд. Среди атрибутов сервиса валидатор (обрабатывает имена пользователей), два репозитория (взаимодействие с таблицами Player и Match), а также кеш. 

Кроме методов создания матча и обновления счета сервис содержит методы создания Dto для передачи в контроллер и последующей обработки в шаблонах html (в том числе выдача списка всех матчей с учетом фильтров). 

### **Repository, Model, Data**

В модуле `repository` помещаются классы для обработки запросов в бд по поиску и созданию игроков, матчей, обновлению счета, выдаче списка завершенных и незавершенных матчей. В модуле `model` описываются классы таблиц бд. Сама бд и методы взаимодействия с ней (подключение сессии) находятся в модуле `data` главного каталога проекта.

## 🧠 Ключевые архитектурные решения

### ⚡ Кэш активных матчей
- Незавершенные матчи хранятся в памяти (`MatchCache`)
- Обновление счета происходит без записи в БД
- Сохранение в БД только при завершении матча или остановке сервера

### 🎾 Логика теннисного счета
- Вынесена в отдельный `ScoreMixin`
- Внутри использует DTO (`ScoreDto`) для удобства
- Публичный метод работает со словарями (совместимость с БД)

### 🔍 Фильтрация и пагинация
- Вынесены в `FilterMixin`
- Поддержка фильтрации по имени игрока
- Фильтр "только завершенные матчи"

## 🧪 Тестирование

Тесты покрывают основные сценарии обновления счета:

- Обычные очки (0→15→30→40)
- Deuce и Advantage (40-40 → AD)
- Выигрыш гейма
- Выигрыш сета
- Тайбрейк
